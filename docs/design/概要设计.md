# DeepCode 概要设计文档

> 状态:进行中(概要设计阶段)
> 最近更新:2026-06-13

## 1. 项目定位

DeepCode 是一个参考 Claude Code / Codex 打造的 coding agent。

- 实现语言:Python
- 核心框架:LangGraph(作为有状态 agent 的编排引擎),在其上二次开发
- 注意:**克制使用 LangChain 主包的高层抽象**(Chain / Agent executor 等),保留对 agent loop 的高控制权;LLM 调用尽量贴近底层 SDK 或薄封装

## 2. 设计目标与约束

### 2.1 交互形态
- **CLI 为主**:首要交付形态,在终端中交互
- **IDE 插件为辅**:后续接入,通过**复用同一个核心引擎**实现,IDE 层只是一个客户端

### 2.2 模型选型
- **主力模型:DeepSeek**(提供 OpenAI 兼容 API)
- provider 做成抽象层,支持多模型可切换

### 2.3 由 DeepSeek 派生的关键约束(影响全局设计)
| 约束 | 影响 |
|---|---|
| context window 相对小(约 64K 量级,需以官方最新为准,Claude 为 200K) | 上下文管理更关键,不能裸塞文件 |
| agentic / 工具选择能力相对 Claude 偏弱 | "放手让模型自主探索"风险大,需工程补位 |
| function calling 复杂多轮稳定性待验证 | 工具调用可靠性需早期重点压测 |

**核心设计哲学**:用工程帮模型把上下文消化好再喂进去,降低对模型自主探索能力的依赖;而不是照搬 Claude Code "放手让强模型自己 grep" 的打法。

## 4. 系统分层架构

```
┌─────────────────────────────────────────────┐
│  客户端层   CLI(主) / IDE 插件(后续)        │   复用同一核心引擎
├─────────────────────────────────────────────┤
│  核心编排层 (LangGraph)                       │
│   agent loop:感知→决策→调用工具→观察→再决策   │
│   router 节点(意图分类) / checkpoint / 中断   │
├─────────────────────────────────────────────┤
│  工具层                                       │
│   grep/glob · read · edit · bash · symbol_*  │
├─────────────────────────────────────────────┤
│  上下文 / 检索层                              │
│   统一 Retriever 接口                         │
│    ├─ TextRetriever   (grep/glob)   [一期]   │
│    ├─ SymbolRetriever (tree-sitter) [一期]   │
│    └─ EmbeddingRetriever (RAG)      [二期]   │
│   repo-map 大纲注入 · rerank/合并             │
├─────────────────────────────────────────────┤
│  Provider 层                                  │
│   DeepSeek(OpenAI 兼容) / 可切换其他模型     │
└─────────────────────────────────────────────┘
```

## 5. 上下文与检索子系统(重点)

### 5.1 统一 Retriever 接口
所有检索方式实现同一接口:输入 `query`,输出"代码片段 + 来源位置(文件/行号范围)"。三种实现可插拔,新增实现不返工。

### 5.2 三种检索实现对比
| 实现 | 底座 | 擅长 | 实现成本 | 阶段 |
|---|---|---|---|---|
| TextRetriever | grep/glob | 字面文本搜索、精确读取 | 低 | 一期 |
| SymbolRetriever | tree-sitter | 结构性查询(定义在哪/被谁调) | 中 | 一期 |
| EmbeddingRetriever | 向量库 + embedding | 概念性/语义模糊查询 | 高 | 二期(按需) |

### 5.3 repo-map 大纲注入
默认用 tree-sitter 抽取"文件 → 类/函数签名"清单注入 context,作为模型的结构先验,给弱 agent 兜底。

### 5.4 检索路由策略(D7 / D8 / D9)

**主力机制:用户驱动的显式路由(`/rag` 命令)**
- 概念性/语义模糊查询 → 用户显式输入 `/rag <问题>` → 走 EmbeddingRetriever
- **结果回灌方式(D10)**:检索片段作为上下文并入主消息流,agent 在**完整 loop 内**基于它继续作答;不走独立短链路,以保证概念性问题与前后对话、后续代码操作的上下文一致性
- 其余查询(结构性、精确读取)走默认流程:模型在工具层选择 grep / symbol / read,辅以 repo-map 大纲先验
- 设计动机:用户是最准的"意图分类器"。把"选检索方式"这个决策交给人,直接删除自动意图分类这一最不可靠环节,正面规避 DeepSeek 自主决策弱的短板;同时实现最简、无额外路由 LLM 成本。

**`/rag` 的边界(已知局限,记录在案)**
- 仅覆盖**用户主动发起**的概念性查询;**覆盖不了 agent 在循环深处自主产生的语义检索需求**(`/rag` 是入口命令,非 agent 可自调)。一期可接受。
- 后续若需 agent 循环内自主语义检索,可再将 embedding 暴露为 agent 工具,或启用下方自动路由 —— 两者与 `/rag` 不冲突。

**未来可选增强(D8,非一期/二期必选)**
- 自动意图 router 节点 + hybrid(并行多 retriever + rerank 合并),用于补齐"循环内自主语义检索"。
- 引入时需评估路由器自身准确率(规则/关键词 或 廉价 LLM 分类)。

**命令子系统(D9)**
- `/rag` 是 slash command 子系统的首个成员;CLI 层实现通用命令解析,不 hardcode 单个命令,便于后续扩展更多命令。

### 5.5 Embedding 新鲜度方案(应对高频增删改,二期落地)
- **按结构切块**:tree-sitter 在函数/类边界切,改一个函数只失效一个 chunk
- **chunk 内容 hash 增量**:只重 embedding 变化的 chunk
- **Merkle 树文件级粗筛**:快速定位哪些文件改了
- **命中后重读源码**:索引只负责"带到大概位置",真实代码实时从磁盘读 → 容忍索引轻度陈旧
- **debounce**:保存时/空闲时异步建索引,不在每次按键时建

## 6. 模型接入(Provider 层)

- DeepSeek 走 OpenAI 兼容 API:用 `openai` SDK 或 `langchain-openai` 改 `base_url`
- provider 抽象:统一封装 chat / tool-calling,运行时可切换模型
- **风险项**:DeepSeek function calling 在复杂多轮 agentic 循环下的稳定性,需一期重点压测(是否按 schema 返回、是否漏调用、参数是否正确)
- **推理 token 处理(一期,D35)**:接 DeepSeek-R1 等推理模型时,provider 需识别并剥离/单独呈现 reasoning token(`reasoning_content` / `<think>`),不让其污染消息历史与工具解析(对标 Aider `reasoning_tags`)
- **effort 映射(一期,D36)**:provider 把 `/effort` 的等级翻译成具体推理控制 —— 推理模型映射到思考预算 / `reasoning_effort` 参数;不支持预算的模型降级为"思考深度"行为提示 + 规划深度
- **多模态(二期,D27)**:provider 接口需支持多模态消息(image content),二期接入视觉模型(DeepSeek-VL 或其他)以支撑图像输入
- **Architect 双模型(二期,D26)**:provider 抽象需支持同时配置"规划模型"与"执行模型"

## 7. 编排层设计(LangGraph,D11)

### 7.1 State 结构
- `messages`:主消息流(assistant 的 tool_calls、tool 执行结果、`/rag` 检索结果均并入此处,D10)
- `cwd` / `repo_root`:工作目录与仓库根
- `retrieved_context`:检索结果结构化暂存(便于 rerank / 去重)
- `iteration` / `budget`:轮次计数与 token/步数预算
- `plan`:可选任务分解(二期启用,D16)

### 7.2 节点与边
- `agent` 节点:调 DeepSeek,产出 assistant message(可能带 tool_calls)
- `tool_executor` 节点:执行 tool_calls,结果写回 `messages`;**一轮多个 tool_call 时并行执行(D34)**,写权工具(edit/bash)的并发安全见详细设计
- 条件边:有 tool_calls → `tool_executor` → 回 `agent`;无 → `END`
- slash 命令入口拦截:`/rag` 先检索并入 `messages`,再进 `agent` 循环
- checkpoint:LangGraph 自带 checkpointer,支撑中断恢复

### 7.3 终止条件
模型不再请求工具 / 超最大轮次 / 超预算 / 用户中断。

> 流式输出 + 中途打断(Ctrl-C 打断生成或工具执行)的具体机制留到**编码期**细化(三家均有精细处理),此处不作设计决策。

### 7.4 检查点与会话恢复(D22,一期)
- 用 LangGraph checkpointer 持久化图状态(messages / 进度等)
- 支持中断后恢复:`--resume` 或选择历史会话继续执行
- checkpointer 后端选型(内存 / SQLite)见详细设计

## 8. 工具层与编辑策略

### 8.1 工具集(一期)
`grep/glob` · `read` · `edit` · `bash` · `symbol_outline` / `find_definition` / `find_references`

**实现优先级(D30)**:`bash`(shell)是 P0、最先实现,且与本地 `Executor`(§9)一起落地 —— 因为 git/lint/test/跑代码全经它路由,是地基。其余工具次之。

**工具输出截断(D31,一期)**:`bash`/`read` 等工具的输出统一做字节/行/token 上限截断,超出时返回截断版 + 提示(可用 `offset`/`grep` 续看)。避免大输出撑爆 DeepSeek 的小 context。

### 8.2 编辑策略(D12 / D32)
- 主:精确字符串替换(`old_string` + `new_string`)—— 对弱模型最稳、token 省
- 兜底:新建文件 / 小文件用整文件写入
- 不采用 unified diff(弱模型生成易坏)
- **最小鲁棒重试(D32,一期)**:精确匹配失败时,回灌"未匹配 + 最相似代码块/行号",让模型据此重试(对标 Aider `SearchReplaceNoExactMatch`);完整 fuzzy 多策略见四期
- tool description 要写细、给示例(对 DeepSeek 友好,降低选错/格式错)

### 8.3 文件操作与命令执行(本地执行,D13)
- `read`/`edit` 等文件操作直接作用于本地工作区
- `bash` 命令在本地子进程执行,受权限确认(D17)约束
- 取消原 Docker 的 host/容器割裂方案;隔离需求由四期 OS 级沙箱满足

### 8.4 MCP 集成(D21,一期)
- DeepCode 作为 **MCP client**,连接外部 MCP server,导入其 tools / resources
- 工具层统一:内置工具与 MCP 工具用同一接口注册给 agent,对模型透明
- 配置:MCP server 列表在 `.deepcode/` 配置中声明

### 8.5 Skill 系统(D20,一期)
- skill = **可复用能力模块**:打包的指令/工作流,可选附带专用工具
- agent 可发现并按需调用(对标 Claude Code skill 机制)
- 一期交付基础的 skill 加载与调用机制;skill 格式/清单见详细设计

## 9. 执行模型(D13,修订)

- 方案:**默认本地子进程执行**(一期),不套容器
- 执行层抽象成 `Executor` 接口,本地执行为首个实现,保留后续插入隔离实现(四期 OS 级沙箱)的可能
- **取消 Docker 的原因**:三家参考无一用强制容器(Aider/Claude Code 本地执行、Codex 走 OS 级沙箱);Docker 带来镜像/挂载/启动开销与 host-容器文件割裂,对一期地基过重
- 安全:写/执行命令走权限确认(§9.1);这是无默认沙箱时的主要安全屏障
- 隔离的强需求由**四期 OS 级轻沙箱**(Landlock/seccomp/Seatbelt,对标 Codex)满足

### 9.1 权限模型(D17)
- 只读操作(read/grep/glob/symbol_*)自动放行
- 写/执行操作(edit/bash)需用户确认,支持"本会话记住"与允许列表
- 提供可选 yolo 全自动模式(无沙箱时尤需谨慎;OS 级沙箱见四期)

## 10. CLI 层(D14)

- 渲染:Rich(markdown / 代码高亮 / 流式输出)
- 输入:prompt_toolkit(历史、多行、补全)
- slash command 子系统:通用命令解析,首成员 `/rag`(D9)
- 复用核心引擎;IDE 插件后续作为另一客户端接入

### 10.1 `/effort` 命令(D36,一期,对标 Claude Code)
- 用户旋钮:`/effort [low|medium|high|max|auto]`,控制模型的"用力程度"
- 语义(参考 Claude Code):low=快速最小、medium=平衡、high=全面、max=最深推理、auto=按模型默认
- 实现:命令只负责设置当前 effort 等级(可持久化到配置 D18);**真正的翻译在 provider 层(§6,D36)** —— 等级 → 推理预算 / `reasoning_effort`,或降级为思考深度行为提示 + 规划深度
- 等级随会话保持,影响后续每轮 `agent` 节点调用

## 11. 上下文管理与会话记忆(D15)

- 策略:滑动窗口 + 自动摘要压缩
- 接近 context 上限时,将较早对话摘要压缩,保留近期原文
- 跨会话持久化到磁盘(可恢复历史会话)
- 与 DeepSeek 小 context 强相关,一期必须项

### 11.1 项目记忆/规则文件(D25,一期)
- 加载类 CLAUDE.md / AGENTS.md / .cursorrules 的项目约定文件,注入 system prompt
- 分层:用户级(`~/.deepcode/`)+ 项目级(`.deepcode/` 或仓库根),与配置分层(D18)一致
- 常驻上下文,作为模型的项目级先验

## 12. 配置与密钥管理(D18)

- 配置分层:用户级 `~/.deepcode/`(全局默认)+ 项目级 `.deepcode/`(项目覆盖)
- 密钥:走环境变量(如 `DEEPSEEK_API_KEY`)或独立密钥文件,**不入库**
- 项目级 `.deepcode/` 中的密钥相关文件应进 `.gitignore`

## 13. 分阶段实施计划

- **二期(增强,功能已冻结 D23/D24,实现待一期完成后确定)**:
  - *检索增强*:EmbeddingRetriever、`/rag` 命令(D10)、新鲜度增量索引(Merkle + chunk-hash,§5.5)、(可选)自动 router + hybrid(D8)

  - *机制层(D29)*:**Hooks 生命周期钩子**(PreToolUse / PostToolUse / Stop / UserPromptSubmit / SessionStart 等事件,对标 Claude Code);**Subagents 并行子任务**(AgentTool/Task,对标 Claude Code)
  - *多模态(D27)*:**图像输入**(截图/原型图/报错截图)—— 需 provider 接入视觉模型(DeepSeek-VL 或其他)、支持多模态消息
  - *联网(D28)*:**Web 搜索 + URL fetch**(查最新库文档/报错,对标 Claude Code/Cursor)
- **三期(扩展,对标成熟 agent)**:
  - IDE 插件(复用核心引擎)
  - *编辑安全网/质量回环(D29,作为内置 post-edit hook 跑在二期 Hooks 底座上)*:**Git 自动提交 + `/undo` + `/diff`**;**Lint/Test 自愈回环**(编辑后跑、失败回灌 agent 自修,对标 Aider 的 auto_commit→lint→reflect)

  - 会话 rewind 回退(对标 Cursor)
- **四期(进一步优化,D33,源自 ref review)**:
  - *编辑鲁棒性完整版*:fuzzy 多策略匹配、`apply_patch` 式结构化编辑格式可选(对标 Aider `editblock`/`search_replace`、Codex `apply_patch`)
  - *repo-map 排序*:基于符号引用图的 **PageRank 排序 + 缓存**,让大库下 repo-map 真正有用(对标 Aider `repomap.py` networkx pagerank)
  - *命令安全策略引擎*:解析/规范化命令判定危险等级,替代粗放的"写/执行都问"(对标 Codex `execpolicy`)

  - *上下文压缩精细化*:分段截断、按类型保留、智能 compact(对标 Codex `compact`/`context_manager`)

  - *跨会话持久记忆*:学习型记忆,超出项目规则文件(对标 Codex `memories`)
